# Beet Sprout — Звіт про результати тестування продуктивності API (JSONPlaceholder)

**Ціль тестування:** перевірити базову стабільність та поведінку API під стрес-навантаженням і спробувати визначити *мінімальні комбінації параметрів*, за яких сервіс перестає витримувати навантаження.

- **Target:** `https://jsonplaceholder.typicode.com`
- **Ендпоінт:** `/posts`
- **Методи:** `GET`, `POST`, `PUT`, `PATCH`, `DELETE`
- **Критерій “fail” (точка зламу):**
  - `Error % > 0` (таймаути, 4xx/5xx, мережеві помилки), **або**
  - падіння асертів, **або**
  - явні `connect/response timeout`.

## 1) Загальні параметри тесту

### 1.1. Тест-плани
- **Beet Seed (base):** 1 потік, 3 ітерації, 10 секунд.
- **Beet Sprout (stress):** послідовні прогони зі збільшенням кількості користувачів (threads).

### 1.2. Асерти (валідація)
- **Status code**:
  - `200` для `GET/PUT/PATCH/DELETE`
  - `201` для `POST`
- **Body validation** (перевірка базового вмісту відповіді)

## 2) Результати за етапами тестування

> Нижче наведені результати по 4 етапах навантаження. Основні метрики взято з **Summary Report**, **Aggregate Report**, **Response Time Graph**

### Етап 1: Базове навантаження (Beet Seed)
**Налаштування:** 1 потік, 10s ramp-up, 3 ітерації (очікувано 15 запитів).

**Результати:**
- **Загальна кількість запитів (#Samples):** 15  
- **Error %:** 0.00%  
- **Максимальний час відповіді:** 457 ms (метод **POST**)  

**Висновок:** сервіс працює коректно, всі функціональні асерти пройдено успішно (коди 200/201 + перевірки тіла).

### Етап 2: Стрес-тест (200 користувачів)
**Налаштування:** 200 потоків, 10s ramp-up, тривалість 3 хв.

**Результати:**
- **Загальна кількість запитів:** 67,734  
- **Throughput:** ~99.5 req/sec  

**Висновок:** система демонструє високу пропускну здатність. Перша деградація: час відгуку для методів запису (**POST/PATCH**) починає зростати (наближається до лінійного росту при збільшенні навантаження).


### Етап 3: Розширений стрес-тест (500 користувачів)
**Налаштування:** 500 потоків, тривалість 3 хв.

**Результати:**
- **Загальна кількість запитів:** ~206,000  
- **Throughput:** ~196.9 req/sec  

**Висновок:** за графіками часу відповіді спостерігається нестабільність (“пилка”). Сервер обробляє черги, але затримки стають нерівномірними.


### Етап 4: Екстремальне навантаження (1000 користувачів)
**Налаштування:** 1000 потоків, тривалість 3 хв.

**Результати:**
- **Загальна кількість запитів:** 483,473  
- **Error %:** 0.00%  
- **Пікове навантаження:** 336.4 req/sec  
- **Піки часу відповіді для POST/PATCH:** до ~400 ms  

**Висновок:** навіть при 1000 користувачах сервіс не видав помилок (Error% = 0). Це вказує на високу стійкість інфраструктури, але видно суттєве зростання затримок для запитів модифікації (POST/PATCH).


## 3) Загальний технічний висновок

### Стабільність
- В усіх сценаріях запити пройшли перевірки статус-кодів (200/201), **помилок (Error%) у наданих даних не зафіксовано**.

### Продуктивність
- Найшвидший метод — **GET** (≈30–35 ms у base).
- Найповільніші — методи модифікації даних (**POST**, **PATCH**), чутливі до росту навантаження.

### Межа системи / “точка зламу”
- Для безкоштовного API JSONPlaceholder **точку зламу (Error%>0 або 5xx)** при навантаженні до **1000 потоків** знайти **не вдалося**.
- При цьому зафіксовано деградацію часу відповіді (особливо POST/PATCH) — тобто “межа” проявляється **в затримках**, а не в помилках.
